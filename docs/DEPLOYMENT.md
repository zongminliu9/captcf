# CapTCF ÔÇö Private Beta deployment

This deploys the current baseline as a **real, internet-reachable Private Beta** with a **remote
managed PostgreSQL** (never a local/dev/test DB), HTTPS, secure cookies, migrations, and the
production seed. Payments are disabled; invited testers get Premium via an admin control.

> **One action is yours.** A real deployment needs an account on a host + database. This machine has
> no authenticated deploy/DB account, and creating accounts / entering passwords is something only
> you can do. Everything else ÔÇö config, env manifest, migrations, seed, health checks ÔÇö is prepared
> here and runs automatically. Pick a path below.

---

## Environment manifest

| Variable | Beta value | Notes |
| --- | --- | --- |
| `NODE_ENV` | `production` | enables secure cookies + strict env validation |
| `DATABASE_URL` | managed Postgres URL | **must be remote**, not localhost/dev/test |
| `AUTH_SECRET` | 32+ random chars | `openssl rand -hex 32`; app refuses to boot if weak in prod |
| `APP_URL` | your HTTPS URL | optional on Render (falls back to `RENDER_EXTERNAL_URL`) |
| `ENABLE_DEMO_ACCOUNTS` | `false` | never seed demo/admin accounts in the beta |
| `PAYMENTS_PROVIDER` | `beta` | real checkout disabled; no self-upgrade |
| `AI_FEEDBACK_PROVIDER` | `none` | local rubric only (unofficial) |
| `STORAGE_DRIVER` | `local` | recordings on local disk (ephemeral without a volume) |
| `STORAGE_LOCAL_DIR` | writable path | `/tmp/uploads` (Render) or `/data/uploads` (Docker volume) |
| `NEXT_PUBLIC_BETA` | `true` | shows the "Private Beta" strip + feedback entry |
| `NEXT_PUBLIC_SHOW_DEMO_ACCOUNTS` | `false` | hides demo credentials on the login page |

Secrets are never committed: `.env*` and `.pgdata/` are git-ignored, and `.dockerignore` keeps them
out of the image. `AUTH_SECRET` is generated by the host, not stored in the repo.

## Path A ÔÇö Render Blueprint (recommended, one click)

Best fit: long-running Node server + managed Postgres, no Docker, migrations run in the build env.

**Your one action:**
1. Create/sign in to a Render account at <https://render.com> (GitHub login connects the repo).
2. **New Ôćĺ Blueprint**, choose the `zongminliu9/captcf` repo, and **Apply**.

Render reads [`render.yaml`](../render.yaml) and automatically: provisions PostgreSQL 17, wires
`DATABASE_URL`, generates `AUTH_SECRET`, runs `pnpm build` then `pnpm db:migrate && pnpm db:seed`,
starts the app, and serves it at `https://captcf.onrender.com` (or similar) with a health check on
`/api/health`.

Free-plan caveats (fine for a beta, upgrade for longevity): the web service sleeps when idle
(~1 min cold start) and free Postgres expires after 30 days. Choose the `starter` plans in
`render.yaml` to avoid both.

After it is live, grant yourself admin by promoting your account (see "First admin", below), then
use **Admin Ôćĺ Acc├Ęs Beta** to grant Premium to invited testers.

## Path B ÔÇö Docker (Fly.io / Railway / any container host)

Uses [`Dockerfile`](../Dockerfile). The image runs `db:migrate && db:seed` (both idempotent) then
`pnpm start`. Provide the same env vars from the manifest and attach a managed Postgres.

- **Fly.io:** `fly launch` (detects the Dockerfile) Ôćĺ `fly postgres create` Ôćĺ `fly postgres attach`
  Ôćĺ set the env vars above with `fly secrets set` Ôćĺ `fly deploy`. Mount a volume at `/data` if you
  want recordings to persist. Your one action is `fly auth login`.
- **Railway:** New Project Ôćĺ Deploy from GitHub repo Ôćĺ add a PostgreSQL plugin (sets `DATABASE_URL`)
  Ôćĺ add the env vars Ôćĺ deploy. Your one action is signing in + connecting the repo.

## First admin

Demo/admin accounts are intentionally **not** seeded in production. To make yourself admin:

1. Register normally on the live site.
2. Promote your user in the database once:
   ```sql
   UPDATE users SET role = 'admin' WHERE email = 'you@example.com';
   ```
   (Render: Dashboard Ôćĺ the database Ôćĺ **Connect** Ôćĺ PSQL. Fly: `fly postgres connect`.)

## Database backup & restore

Managed Postgres providers snapshot automatically; you can also do it by hand:

```bash
pg_dump "$DATABASE_URL" > backup.sql          # backup
psql "$DATABASE_URL" < backup.sql             # restore into an empty database
```

The seed is idempotent, so content can always be rebuilt with `pnpm db:migrate && pnpm db:seed`.
Only user-generated data (accounts, attempts, recordings) needs real backups.
